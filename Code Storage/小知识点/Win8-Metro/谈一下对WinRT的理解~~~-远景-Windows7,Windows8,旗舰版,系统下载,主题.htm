WinRT是一个新的API 集合，具有以下特性：
1.它实现了Metro UI规范的UI库
2.为Windows开发人员提供一个简单的UI编程模型，你不需要学习Win32API的那些复杂的API了
3.它使用XAML-base的UI系统
4.API都设计成了异步的
5.它和.NET一样是个沙箱的API，自成体系，用于创建AppStore上的应用程序。
6.API的元数据格式是ECMA335，和.NET一样的标准。

由于本人是.Net的程序员，以Net程序员的角度来看待WinRT。
已有经验都可以继续使用，WinRT虽然本质上和Net框架不是一回事，但其实感觉又差不多，就全当是native的并加入了点新东西的.Net框架用了。
而开发上，我们当然继续用C#，这是0成本，XAML，就像WPF，就等于基本会了siverlight，熟悉下WP7的API和全景视图这些界面模版，直接就可以做开发了，成本忽略不计。
当然，作为.Net程序员，基本都是做B/S的，HTML+Javascript任何一个.Net程序员多少都会，偏重界面和动画可能用XAML，不然界面用HTML对于我们来说更顺手。
至于WinRT，无论是C++，还是C#，还是Javascript，都可以调用，而且WinRT是native的，X86上和ARM上都用同样的实现，对上层是透明的，对于WinRT的调用也就不再区分是X86还是ARM，由于是native的，这其实还解决了.Net效率的问题。
com到Net到WinRT，感觉就是一个具有Net思想的新一代的Win32API。

补充内容 (2011-9-18 00:05):
ARM无法直接运行X86的程序，应该已基本明确了，不过目前看来未来Windows store的应用，你是不需要为X86和ARM开发两个版本的。

其实我现在很好奇一个事情，现在的Windows要兼容Win98甚至更早的程序，已经臃肿不堪了。

而ARM由于反正不能支持运行旧的X86程序，会不会把这些束缚砍掉，只要WinRT就好，只需要和未来Win8的程序兼容就好。

这会让我对ARM版Win8更加期待，因为没有束缚，把这些砍掉，Win8可以瘦身多少了，可以高效多少啊。

ARM版早期程序或许会看起来少一点，但ARM版或许才算是革命性的东西，ARM版早期看来只能娱乐为主了，全能还是得X86，不过软件总会出新版的，而且新版也不必为ARM单独开发，ARM和X86在应用方面的差距会逐渐变小。

补充内容 (2011-9-18 00:33):
